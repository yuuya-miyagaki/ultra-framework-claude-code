# テスト駆動開発 (TDD)

> テストを先に書く。失敗を確認する。最小限のコードで通す。

## Iron Law

```
テストなしのプロダクションコードは禁止
```

テスト前にコードを書いた場合 → 削除してやり直す。

- 「参考として残す」は禁止
- 「テストを書きながら修正する」は禁止
- 削除は削除。テストからやり直す

## RED-GREEN-REFACTOR サイクル

### RED: 失敗するテストを書く

1. 1 つの振る舞いをテストする
2. テスト名は振る舞いを明確に記述する
3. 実コードを使う（モックは必要最小限）

```
テストの要件:
- 1 つの振る舞い
- 明確な名前
- 実コード（モックは避けられない場合のみ）
```

### RED の確認（必須）

```bash
<テストコマンド>
```

確認事項:
- テストが **失敗する**（エラーではなく失敗）
- 失敗メッセージが期待通りである
- 機能がないから失敗する（タイポではない）

テストが通った場合 → 既存の振る舞いをテストしている。テストを修正する。

### RED→GREEN の探索（ReAct）

テストが失敗した状態から GREEN にする際:

1. エラー出力を **読む**（Observe）
2. 原因を **1つ** 推測する（Think）
3. **最小変更** を適用してテスト再実行（Act）

1回の Act で 1つの変更のみ。複数変更を同時に適用しない。

### GREEN: 最小限のコード

テストを通すための最短コードを書く。

- 機能を追加しない
- 他のコードをリファクタリングしない
- テストの範囲を超える「改善」をしない

### GREEN の確認（必須）

```bash
<テストコマンド>
```

確認事項:
- テストが通る
- 他のテストも通る
- エラーや警告がない

テストが失敗した場合 → コードを修正する（テストを修正しない）。

### REFACTOR: クリーンアップ

GREEN の後のみ:
- 重複を除去する
- 名前を改善する
- ヘルパーを抽出する

テストを GREEN に保つ。振る舞いを追加しない。

## 一般的な合理化への反論

| 言い訳 | 現実 |
|---|---|
| 「シンプルすぎてテスト不要」 | シンプルなコードも壊れる。テストは 30 秒 |
| 「後でテストを書く」 | 即座に通るテストは何も証明しない |
| 「手動でテスト済み」 | アドホック ≠ 体系的。記録なし、再実行不可 |
| 「X 時間の作業を削除するのは無駄」 | サンクコスト。検証できないコードは技術的負債 |
| 「参考としてコードを残す」 | それを修正する = テスト後にコードを書いている。削除 |
| 「先に探索が必要」 | 探索は OK。探索結果は捨てて TDD で開始 |
| 「TDD で遅くなる」 | TDD はデバッグより速い |

## Red Flags（即座に停止してやり直し）

- テスト前にコードを書いた
- テスト後に実装した
- テストが即座に通った
- なぜテストが失敗したか説明できない
- テストを「後で」追加した
- 「今回だけは例外」と合理化した

**上記すべて → コードを削除。TDD でやり直す。**

## 検証チェックリスト

作業完了前に確認:

- [ ] 新しい関数 / メソッドごとにテストがある
- [ ] 各テストが失敗するのを確認してから実装した
- [ ] 各テストは機能がないから失敗した（タイポではない）
- [ ] 各テストを通すために最小限のコードを書いた
- [ ] 全テストが通る
- [ ] 出力がクリーン（エラー、警告なし）
- [ ] テストは実コードを使っている（モックは必要最小限）
- [ ] エッジケースとエラーケースがカバーされている

全てチェックできない場合 → TDD をスキップした。やり直す。

## 推奨パターン

### Result 型と TDD

外部入力（ファイル、API レスポンス）を解析する純粋関数には Result 型を推奨:

- `{ ok: true; data: T } | { ok: false; error: string }`
- try/catch で例外を吸収し、呼び出し側に型安全なエラーハンドリングを強制
- テストが書きやすい（ok: true/false の分岐を明示的にテスト）
