# 用語集

<!-- exit-check: 主要用語が網羅・定義が曖昧でない・クライアント確認済み → open-questions へ -->

クライアントが使用する用語とその定義を記録する。認識齟齬の防止が目的。

## 用語テーブル

| クライアント用語 | 定義 | 使用コンテキスト |
|----------------|------|----------------|
| gate | フェーズ間の必須承認チェックポイント。ユーザーの明示的承認がなければ次フェーズに進めない | Client/Dev 両モードのフェーズ遷移時 |
| phase | Client または Dev ワークフロー内の 1 ステージ。各フェーズには固有の artifact と exit-check がある | ステートマシン全体 |
| mode | Client（要件定義側）または Dev（開発側）の運用コンテキスト。ハードゲートで遷移を制御する | ステートマシンのトップレベル分類 |
| hook | Claude Code の PreToolUse/PostToolUse で実行される shell スクリプト。PaC を実現する手段 | .claude/hooks/ 配下の bash ファイル |
| PaC (Policy as Code) | ドキュメントの記述ではなく hook スクリプトで強制するルール体系。違反時はツール実行を deny する | フレームワーク設計原則 |
| artifact | フェーズ完了に必要な成果物（ドキュメントまたはコード）。evidence-based completion の根拠となる | 各フェーズの exit-check |
| translation mapping | クライアント用語を機能的意味と実装ヒントに変換する 3 層変換テーブル | Client モードの handover 前に必須 artifact として作成 |
| scaffold | bin/setup.sh によるプロジェクト初期構築。プロファイルに基づきテンプレートからファイルを生成する | フレームワーク導入時 |
| thin framework | CLAUDE.md を最小限に保つ設計原則。700 語上限を維持し、詳細はルールファイルやスキルに分散する | フレームワーク設計全体 |
| contract validator | check_framework_contract.py による構造・内容の自動検証。プレースホルダ残存やファイル欠損を検出する | CI およびレビュー時 |
| exit-check | フェーズ完了時に確認すべき条件リスト。各ドキュメントの HTML コメントとして埋め込まれる | 全テンプレートファイル |
| STATUS.md | セッション間の状態管理ファイル。現在フェーズ、ゲート状態、参照先、ブロッカーを記録する | 全セッションの開始・終了時 |
